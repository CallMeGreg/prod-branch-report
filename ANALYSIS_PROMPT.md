# Production Branch Analysis Prompt

Use this prompt with an LLM (e.g., GitHub Copilot Chat, ChatGPT, Claude) to analyze the CSV report generated by `prod-branch-report`. Paste the CSV content after this prompt, or attach the file.

---

## Prompt

You are an expert at analyzing GitHub repository branching strategies. You will receive a CSV report containing deterministic signal data for repositories in a GitHub organization. Each row is a repository and each column is a signal that may indicate which branch(es) serve as "production" branches.

### Columns explained

| Column | Meaning |
|--------|---------|
| Repository | Full `org/repo` name |
| Default Branch | The repo's configured default branch |
| Protected Branches | Branch name patterns with branch protection rules applied |
| Ruleset Target Branches | Branches targeted by active repo-level branch rulesets |
| Deployment Branches (prod) | Branches that have been deployed to a "production" environment |
| Release Target Branches | Branches that releases were created from (`target_commitish`) |
| Tagged Branches (by count) | Branches with the most release tags, ordered by count |
| Top PR Merge Target | Branch receiving the most merged pull requests (with count) |
| Workflow Push Branches | Branches listed in `on.push.branches` in GitHub Actions workflows |
| Most Active Branch (6mo) | Branch with the highest commit count in the last 6 months |
| Deepest Branch (total commits) | Branch with the most total commits (longest history) |

### Your task

For each repository, produce a hypothesis with these fields:

1. **Repository**: The repo name
2. **Multiple Production Branches?**: `Yes` or `No` — whether the repo likely maintains multiple simultaneous production branches (e.g., `release/1.x` and `release/2.x` both receiving deploys/tags)
3. **Candidate Production Branch(es)**: The branch name(s) most likely serving as production, ordered by likelihood
4. **Confidence**: `high`, `medium`, or `low`
5. **Reasoning**: 1-2 sentences explaining your conclusion

### Guidelines

- **High confidence** = 3+ signals converge on the same branch(es)
- **Medium confidence** = 2 signals agree, or some ambiguity
- **Low confidence** = sparse data or conflicting signals
- The default branch is often but NOT always the production branch
- Multiple production branches = evidence that the repo actively maintains >1 long-lived branch each independently serving production (e.g., both getting deploys, tags, or releases). Patterns like `release/*` in rulesets alone do NOT prove this.
- Empty columns mean no data was found for that signal — this is common and doesn't mean the signal is negative
- When in doubt, bias toward the default branch as the most likely candidate

### Output format

Return a CSV with these columns:
```
Repository,Multiple Production Branches?,Candidate Production Branch(es),Confidence,Reasoning
```
